# 🧬 PROJECT 2100 AI 어시스턴트 강제 룰셋 (Assistant Instruction Rules)

이 파일은 AI 코딩 어시스턴트가 본 프로젝트 워크스페이스를 분석하거나 작업을 시작할 때 **컨텍스트에 자동으로 즉시 반영하고 준수해야 하는 최우선 절대 규칙**입니다.

## 0. Hard Constraints (절대 규칙)

### 0.1 Destructive 작업 사전 승인
- 라이브 서비스 게임의 운영 DB(유저 세이브/재화/랭킹) 접근·수정은 반드시 사전 승인.
- 결제/인앱구매(IAP), 푸시 알림, 외부 백엔드(BaaS) 호출은 사전 승인.

### 0.2 비밀 정보 보호
- 서명 키스토어(.keystore, .p12), 프로비저닝 프로파일, API 키, 서버 시크릿을 출력·로그에 남기지 않음. .env.example만 참조.

### 0.3 바이너리·에셋 보호
- 텍스처, 모델, 오디오, .prefab, .scene, .uasset 등 바이너리/직렬화 에셋은 사용자 확인 없이 수정하지 않음.
- Unity .meta 파일의 GUID, Unreal redirector를 임의로 건드리지 않음 (참조 깨짐 방지).

### 0.4 요청 없는 파일 생성 금지
- 요청되지 않은 README, 설정 파일, 샘플 씬/프리팹을 선제적으로 만들지 않음.

### 0.5 🚨 [핵심/1순위] 주인공 요원 방향성 규칙 (Dual View: Facing Right & Front View)
- 본 게임은 씬에 따라 듀얼 뷰 캐릭터 조립을 사용합니다.
  - **전투 씬 (횡스크롤)**: 요원은 무조건 왼쪽에서 오른쪽을 바라보는 방향(Facing Right)으로 렌더링 및 조립됩니다. 오른팔은 전면에 노출되고, 왼팔은 몸통 뒤(Z-index 조절)에 배치되나 독립 파츠로 조립합니다.
  - **비전투 씬 (로비/대기)**: 요원은 정면(Front View)을 바라보는 구도로 렌더링 및 조립됩니다. 양팔과 양다리가 좌우 대칭으로 완전하게 노출되는 구도입니다.
- **리소스 생성**: 2D 도트 스타일의 전투용 우향(Right-facing) 파츠 세트와 비전투용 정면(Front-facing) 파츠 세트를 각각 독립적으로 생성하여 관리합니다.

### 0.6 🎨 게임 콘셉트 및 애니메이션 규격 (Retro Dot & 10-Frame Sprite)
- **비주얼 스타일**: 조선 시대를 배경으로 하는 레트로풍 **2D 도트(픽셀 아트) 그래픽** 기반이며, 전체적으로 **아기자기하고 발랄한 느낌**을 가진 가벼운 캐주얼 게임 콘셉트를 지향합니다.
- **애니메이션 프레임**: 캐릭터 애니메이션은 **2D 도트 일러스트 스프라이트 10프레임** 규격으로 통일하여 가볍고 경쾌한 동작을 구현합니다.

### 0.7 🎨 AI 리소스 생성 및 조립 방식 규격
- **AI 이미지 생성**: 프레임 단위의 스프라이트 시트 생성을 지양하고, **단일 파츠 세트(머리, 몸통, 손, 발 1장씩)를 고화질 2D 도트 스타일로 생성**하여 에셋으로 사용합니다.
- **애니메이션 구현**: 생성된 2D 도트 단일 파츠들을 코드나 게임 엔진 상에서 **10프레임 규격의 정적 키프레임 위치 배열(프레임 바이 프레임 방식 연출)** 또는 실시간 연산(스켈레탈 방식 연출)을 통해 움직이도록 구현하여 리소스 일관성을 보장하고 개발 비용(토큰)을 절약합니다.
- **리소스 포맷**: 게임 내 모든 리소스(게임 디자인, 일러스트, 아이콘 등)는 SVG(벡터) 방식을 배제하고, 일반 **2D 도트 스타일의 비트맵 일러스트(PNG/JPG)** 포맷으로 일관되게 제작합니다.

## 1. Context Engineering (컨텍스트 관리)

### 1.1 세션 시작 시 참조 순서
1. 프로젝트 루트의 GEMINI.md (존재 시)
2. 엔진/빌드 설정 — ProjectSettings/, *.uproject, project.godot, package.json(JS 게임)
3. 작업과 직접 관련된 스크립트/씬만 선택적으로 읽기
4. 필요 시 아키텍처 문서 (게임 루프, 상태 머신, ECS 구조 등)

### 1.2 Read before Write
- 수정 전 반드시 Read로 해당 파일을 먼저 읽음.
- 존재한다고 가정하는 컴포넌트/매니저/싱글톤은 grep으로 실재 여부 확인.

### 1.3 탐색 전략: Glob → Grep → Read
- `**/*.cs`, `**/*.cpp`, `**/*.gd`, `**/*.js` 등 패턴으로 좁히고, 키워드 grep 후 필요한 부분만 읽기.
- 전체 디렉토리 덤프나 대용량 직렬화 파일 통째로 읽기 금지.

### 1.4 토큰 예산 최적화
- 로그(Player.log, Editor.log)는 tail/grep으로 필요한 부분만.
- 자동 생성된 거대 파일(씬 YAML, 빌드 로그)은 통째로 읽지 않음.

### 1.5 의도 재확인
- 핵심 게임 루프, 물리, 네트워킹 동기화 같은 구조적 결정에서 초기 설계를 벗어나면 멈추고 재확인.

## 2. Execution & Verification Loop (실행 및 검증 루프)

### 2.1 핵심 워크플로우: Explore → Plan → Code → Verify
[Explore] 게임 시스템/요구사항 이해 ➡️ [Plan] 계획 수립 → 사용자 승인 대기 ➡️ [Code] 구현 ➡️ [Verify] 플레이 테스트 / 자동 테스트 ➡️ [Report] 결과 보고

### 2.2 Plan Mode 강제
- 2개 이상 파일 수정 또는 20줄 이상 변경, 혹은 게임플레이 동작이 바뀌는 변경은 구현 전 계획을 먼저 제시하고 승인 대기.
- 계획 포함 항목: (1) 수정 대상 (2) 변경 요지 (3) 예상 사이드이펙트(밸런스/성능/세이브 호환성) (4) 검증 방법.
- 단순 오타·상수 한 줄·명백한 버그는 건너뛸 수 있음.

### 2.3 게임 코드 검증 우선순위
1. 자동 테스트 — Unity Test Framework(EditMode/PlayMode), 단위 테스트(순수 로직)
2. 컴파일/빌드 — 엔진 콘솔 0 에러, 셰이더 컴파일 통과
3. 플레이 테스트 — 해당 기능 동작 직접 확인 (재현 절차 명시)
4. 프로파일링 — 프레임타임, GC alloc, 드로우콜, 메모리(성능 관련 변경 시)
5. 위가 불가능할 때만 정적 리뷰로 대체하고 "실행 검증 불가" 명시.

### 2.4 검증 실패 시
- 원인 기록 후 최대 3회 재시도. 3회 실패 시 현황·가설 보고하고 방향 결정 요청.
- 임시 try/catch로 에러 삼키기, 테스트 비활성화, 경고 무시 우회 금지.

## 3. Constraints & Safety (제약 및 안전)

### 3.1 Deny by Default (승인 후 실행)
- rm -rf, 대규모 에셋 재임포트, 라이브러리 캐시 강제 삭제.
- 엔진/SDK 메이저 버전 업그레이드, 패키지 매니저 lock 전면 갱신.
- 운영 백엔드/매치메이킹/리더보드 서버 호출.
- 전역 환경 설정 변경.

### 3.2 Allowlist (자동 실행 가능)
- 읽기 전용: ls, grep, find, 파일 조회.
- 테스트/빌드 검증: 단위 테스트 실행, 컴파일 체크, 린트.

### 3.3 의존성 관리
- 엔진 플러그인/패키지 버전 고정. 임의 범위 설치 지양.

### 3.4 코딩 컨벤션
- 프로젝트 기존 스타일 우선 관찰 후 일치.
- 엔진 관용 패턴 우선.
- 하드코딩 지양 — 밸런스 수치(데미지, 쿨다운, 가격)는 ScriptableObject/데이터 테이블/설정 파일로. 매직 넘버는 명명된 상수로.
- 성능 주의 — Update() 내 할당, GetComponent/Find 반복 호출, LINQ 남용, 매 프레임 GC alloc 회피.
- 임시방편 주석 지양. 불가피하면 이슈 번호와 함께.

### 3.5 파일 조작 원칙
- 편집은 Edit 우선, 신규 생성은 꼭 필요할 때만.
- 직렬화 에셋/씬 파일은 가능하면 엔진 에디터를 통해 수정하도록 안내하고, 직접 편집 시 사용자 확인.

### 3.6 산출물 저장
- 모든 보고서·분석 산출물은 프로젝트 루트 산출물 폴더(`/docs/` 등)에 저장.

## 4. Bias Mitigation (자기 편향 방지)

### 4.1 완료 선언 전 확인
- [ ] 테스트가 실제로 통과했는가? (출력 로그 확인)
- [ ] 컴파일/셰이더 에러 0건인가?
- [ ] 플레이 테스트로 해당 동작을 직접 확인했는가?
- [ ] 기존 기능(다른 씬/시스템)을 깨뜨리지 않았는가?
- [ ] 세이브 데이터 호환성을 깨지 않았는가?

### 4.2 지양할 표현
- ❌ "잘 작동할 것 같습니다" ➡️ ✅ "PlayMode 테스트 3건 통과, 인게임 동작 확인"
- ❌ "성능 괜찮을 겁니다" ➡️ ✅ "프로파일러: 프레임타임 16.2ms→14.1ms, GC alloc 0B"

### 4.3 실패 인정
- 검증 실패·미완성 구현은 숨기지 않고 명시 보고.

## 6. Verbosity & Tone (출력 및 어조)
- 불필요한 서두("좋은 질문입니다", "도와드리겠습니다") 금지. 본론부터.
- Show, don't tell — 코드를 보여주고 장황한 설명 지양.
- 길이 매칭 — 한 줄 질문엔 한 줄 답변.
- 코드/로그/주석에 이모지 사용 금지 (대화는 허용).
- 아첨 금지.
- 사용자가 사실·기술 판단에서 틀리면 근거와 함께 반박.
- 모르면 "모릅니다"로 명시. 추측으로 메우지 않음.
- 초기 범위를 벗어난 작업이 누적되면 멈추고 현황 정리 후 재확인.
- **답변 스타일 규격**: 유저에게 대답할 때는 장황한 서술을 전면 금지하며 가능한 아주 짧게 요약해야 하고, 정보 제시 시에는 표(Table)를 활용하여 극단적인 가독성을 구현해야 합니다.

- 주석은 프로젝트 기존 주석 언어를 따름.
